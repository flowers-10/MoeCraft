# API 依赖故障手册

## 通用流程

1. 通过 `requestId` 和 `traceId` 确认影响范围、开始时间和最近发布。
2. 禁止直接修改订单、支付、库存或审计表来“修复”状态。
3. 先停止新副作用，再恢复依赖，最后按幂等接口重放。
4. 记录时间线、操作者、重放范围和校验结果；P1 事故完成复盘。

## 数据库不可用

- `/health` 仍可用于确认进程存活；`/readiness` 应返回 503，使负载均衡停止接流量。
- 检查连接数、磁盘、复制延迟和迁移状态；不得通过关闭外键或跳过迁移恢复服务。
- 数据库恢复后先验证只读查询，再开放写流量，并核对事故窗口内的重试请求。

## 支付回调积压

- 暂停自动退款、结算和依赖支付事实的订单推进。
- 按渠道事件 ID 去重，核对签名、金额、币种和原始事件，再按时间顺序重放。
- 比较支付单、渠道账单和订单状态；任何差异进入人工队列，不静默改账。

## 超卖或库存不一致

- 暂停受影响 SKU 销售，保留库存流水和订单快照。
- 以不可变流水重算 `onHand/reserved/available`，定位缺少的锁定、扣减或释放事件。
- 补偿操作使用独立事件和审计记录；不得覆盖历史流水。

## 队列停摆

- 暂停产生不可控积压的入口，记录队列深度、最老任务时间和失败类型。
- 恢复消费者后按幂等键重试；死信只允许高权限操作者检查并重放。
- 核对订单超时、库存释放、通知和导出任务是否发生重复副作用。

## 对象存储不可用

- 保留文件元数据为待处理状态，禁止把未扫描文件标记为可用。
- 禁用新的签名上传/下载，现有业务表不得写入伪造对象键。
- 恢复后验证对象存在性、大小、哈希和扫描状态，再开放访问。
