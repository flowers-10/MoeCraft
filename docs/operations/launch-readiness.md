## 容量基线

本仓库提供仅允许 loopback 的 smoke 工具，避免误压生产：

```bash
pnpm capacity:smoke
TARGET_URL=http://127.0.0.1:3102/readiness pnpm capacity:smoke
```

本地记录（macOS 15.3.1 arm64、Node 24.14.0、G40 working tree）：`/health` 50 请求、并发 10，0 失败，P50 4.65ms、P95 20.81ms、P99 37.64ms、883.07 req/s。该结果只验证脚本和存活端点；500 请求测试在第 60 次后触发全局限流并返回 429，说明单 IP 压测不能代表服务容量。

记录机器、Node 版本、commit SHA、目标路径、请求数/并发、p50/p95/p99、吞吐、错误数和数据库状态。`/health` 只代表进程存活，不代表商品读取或交易能力；staging 还要分别测公开目录、登录、商品详情、购物车、创建订单和支付回调，并使用获批的多客户端流量模型，不能关闭线上安全限流来制造结果。订单/支付模块接入前不能把本次 health 结果当作商城容量结论。

建议 MVP 目标：公共读接口 p95 < 500ms、写接口 p95 < 1000ms、5xx < 1%；支付回调积压和库存锁定失败必须有独立阈值。超过阈值先停止扩大流量，保留 trace/request id 和版本信息。

## 灰度与回滚

1. `internal`：只允许工程/审核账号，验证 `/health`、`/readiness`、登录、RBAC、商品审核和日志脱敏。
2. `canary-merchant`：邀请少量非生产敏感商家，使用 staging provider；观察 15 分钟错误率、P95、readiness 和审计事件。
3. `5% → 25% → 50% → 100%`：每级至少一个观测窗口，窗口内无 P1/P2、无支付/库存事实差异、无跨商家访问后才晋级。

触发回滚：readiness 连续 3 次失败、5xx 五分钟超过 2%、登录失败显著偏离基线、任何权限越权、订单/支付/库存事实不一致或迁移错误。回滚负责人先确认旧镜像与 schema 兼容，再切回 immutable digest；不可通过删除列或修改历史迁移“回滚”。支付/订单异常时先暂停副作用入口，再按事件幂等键处理积压。

## 客服 SOP

- 首次响应只收集 case id、用户可验证信息和问题时间，不索取密码、token、完整支付号或不必要证件。
- `P1`：账号大面积不可用、支付/订单事实异常、疑似数据泄露；立即通知事故负责人和安全负责人，冻结相关副作用。
- `P2`：单商家无法发货/售后、重复通知、性能显著退化；记录 requestId/traceId、影响订单并升级对应模块 owner。
- `P3`：文案、目录、一般咨询；按模板回答并关联协议/售后规则。
- 客服只能查看最小订单/售后字段，退款、补偿、改账和解冻必须走授权后台并写审计；敏感证据放受控工单附件，不进聊天转发。

## 财务对账 SOP（支付模块接入后启用）

每日下载渠道账单并校验文件 hash、日期、币种和总额；按渠道事件号与内部 payment/order/refund 逐笔匹配。差异分为金额、状态、缺单、重复和时间窗口，进入不可变差异表；禁止直接改订单或“用下一单冲平”。人工调整需财务复核 + 平台管理员双确认、备注证据和审计事件。结算前复核退款窗口、佣金快照、商家应付额和异常冻结，生成可重放报告。

## 事故角色与通讯录

仓库只定义职责，不保存个人电话或真实密钥。上线前在内部 on-call 系统填入：Incident Commander、API/数据库 owner、安全/隐私 owner、支付/财务 owner、客服/公关 owner、Release/Rollback decision owner、备份恢复执行人。每季度演练一次，值班接收渠道和升级时限必须可验证；没有已确认的联系人，发布状态保持 blocked。

## RPO/RTO 目标

初始目标（待 staging 演练确认）：交易事实 RPO ≤ 15 分钟、API RTO ≤ 60 分钟、审计日志 RPO ≤ 5 分钟。真实值取决于 MySQL 备份/复制、对象存储版本化和 provider 合同；演练记录必须给出实际恢复时间、数据缺口和改进项。
